## 背景

Choco360 已具備完整的 360 同儕回饋流程。Phase 1 的目標是將「主管考核 (PRP)」資料帶入系統，融合 PRP + 360 回饋，產出有意義的 Superpowers 與 Opportunities。
這份問卷設計得非常好！現在我完全理解了你的維度體系。讓我做一個完整的映射分析：

### 你的 360 維度 vs HR 的 PRP 核心職能 — 完整對照

| 你的 Choco360 維度 | 子題數 | HR 的 PRP 核心職能能對應到？ | 映射品質 |
|---|---|---|---|
| **1. Team Work 與協作** (心理安全感/溝通品質/補位/回饋) | 4 題 | 團隊合作 ✅ + 溝通技巧 ✅ | 🟢 直接對應，但你的更精細 |

### 我的結論

你的維度設計**遠優於** HR 的版本。幾個關鍵觀察：

1. **你的更有「行為錨定 (Behavioral Anchor)」**：每個子題都有具體的行為描述（例如「不僅僅是把程式碼寫完 Done，而是確保功能上線後能穩定運作」），而 HR 的只有一個抽象名詞（「追求卓越」）。
2. **「產品思維」和「工匠精神」是你的獨門武器**：HR 的核心職能完全沒有對應的概念。這兩個維度是軟體團隊最需要被評量的，PRP 那邊反而是空白。
3. **Mark 的 PRP 案例驗證了映射**：他主管在 KPI 3 強調「產品思維」→ 這會直接映射到你的維度 2；主管在核心職能「追求卓越」中描述的行為 → 這會映射到你的維度 3 和 4。

## 一、能力框架分析 — HR 的 vs 你的，用哪個？

### 1.1 兩套框架比較

| | HR 公司制度 (PRP 核心職能) | Abraham 的 Choco360 (360 問卷維度) |
|---|---|---|
| **來源** | 公司 HR 統一訂定 | 你自己針對軟體團隊設計 |
| **項目** | 團隊合作、創新精神、追求卓越、溝通技巧、解決能力 | Team Work 與協作（+ 可在 Admin 自由增刪的維度）|
| **實際使用維度** | 員工每期自選 2 項 | 團隊合作、問題解決、溝通表達、開放回饋 |
| **評量方式** | 員工自述 + 主管文字評語 + 數字打分 (0-100) | 量化 1-5 分 + 質性文字描述（匿名同儕） |
| **評估角度** | 上對下（主管單一視角） | 多角度（自評 + 多位同儕） |
| **適用範圍** | 全公司所有職種 | 你的團隊為主 |

### 1.2 我的建議：以 Choco360 的維度作為「共通語言 (Common Language)」

> [!IMPORTANT]
> 不需要二選一。兩套資料都保留，但用你自己定義的 Choco360 維度作為 AI 合成時的「對齊框架」。

**理由：**
1. HR 的 5 項核心職能太籠統（例如「追求卓越」是一個很模糊的概念），你已經在 Choco360 問卷裡做了更細緻的重新定義。
2. 360 回饋的量化分數（1-5分）是建立在你的維度上。如果 Superpower 要交叉驗證，就應該以有量化數據的維度為主軸。
3. PRP 中的核心職能（如「團隊合作」）可以**語意映射**到 Choco360 的維度（也叫「團隊合作」），AI 在合成時做自然語言對齊即可，不需要在系統中硬綁。

**具體做法：**
- PRP 資料原封不動匯入（保留 HR 原始的 KPI + 核心職能架構）。
- Choco360 的 360 問卷維度不需要改。
- AI 合成時，Prompt 中同時給出兩套框架，讓 AI 做語意對齊：

```
你的維度框架 (Choco360 360 回饋使用的維度)：
1. 團隊合作 — 評估心理安全感與跨團隊補位能力
2. 問題解決 — 評估分析與解決問題的能力
3. 溝通表達 — 評估口頭與書面溝通的清晰度
4. 開放回饋 — 蒐集自由文字回饋（純質性）
5. (管理員未來可動態新增更多維度)

PRP 使用的核心職能（公司 HR 定義，參考用）：
團隊合作、創新精神、追求卓越、溝通技巧、解決能力

→ AI 任務：將 PRP 中的核心職能評語「映射」到 Choco360 的維度上，
  並結合 360 量化分數做交叉驗證，產出 Superpowers。
```

> [!TIP]
> 這代表你**不需要額外建立 `core_competencies` 資料表**。問卷中的 `dimensions` 本身就已經是你的能力框架。未來如果你想擴充維度（例如加入「技術深度」、「領導力」），直接在 Admin Panel 的問卷編輯器中新增維度即可。

## 二、Markdown 匯入可行性分析

### 2.1 格式分析

我仔細讀了 Mark 的 PRP Markdown，以下是解析挑戰：

**✅ 可解析的部分：**
- 基本資訊列（第5行）：`部門 | | 研發部 | 姓名 | 方信登 | 員編 | F25509 | 職稱 | Senior Backend Engineer`
- KPI/核心職能的標記：第一欄 `KPI 1`、`KPI 2`、`核心 職能 1` 等
- 分數：獨立在第10欄的數字（`84`、`89`、`85`）
- 主管評語：位於第6欄的大段文字
- 等第：`□S ☑A □B □C □D` 格式

**⚠️ 解析困難的部分：**
- **合併儲存格**：每個 KPI 實際佔 2~3 列（原主管意見一列 + 新主管意見一列），列之間用空白欄位表示「屬於上一個 KPI」。
- **縱向分隔不一致**：第一列的欄位（部門/姓名/員編/職稱）跟 KPI 列的欄位意義完全不同，但共用同一個 14 欄表格。
- **「新主管意見」位置不固定**：它出現在原主管評語的下一列，靠第6欄中的文字 `新主管意見` 來辨識。
- **特殊字元**：有 `\>` 轉義、`\u000b` (垂直 tab)、`☑` 等需要清理。

### 2.2 推薦方案：AI 輔助解析 (AI-Assisted Parsing)

> [!IMPORTANT]
> 用正則硬解析這份 Markdown 太脆弱（因為每個人的 KPI 數量不同、核心職能項數不同），而且不同人的 Google Doc 格式可能有微小差異。**最穩健的做法是讓 Gemini AI 來解析。**

**流程設計：**

```
使用者上傳 .md 檔案
       ↓
前端讀取檔案內容（純文字）
       ↓
呼叫 Gemini API，Prompt：
  「以下是一份績效考核 Markdown 表格，
    請萃取出以下結構化資料：
    - 基本資訊 (部門, 姓名, 員編, 職稱)
    - KPI 項目 (N 個): 標題, 自述, 重要度, 
      原主管評語, 原主管分數, 新主管評語, 新主管分數
    - 核心職能 (N 個): 類別名, 自述, 
      原主管評語, 原主管分數, 新主管評語, 新主管分數
    - 綜合考核: 員工自述, 原主管評語, 新主管評語, 等第
    請以指定的 JSON Schema 回傳。」
       ↓
前端顯示解析結果（結構化預覽）
       ↓ 使用者確認 ✅
存入 prp_records + prp_items
```

**好處：**
- 容錯性高：即使不同員工的 Markdown 格式略有不同，AI 也能理解。
- 開發成本低：不需要寫複雜的正則解析器。
- 可驗證：使用者匯入後會看到結構化預覽，確認正確後才儲存。

**風險：** AI 偶爾可能解析錯誤（例如把分數對到錯誤的 KPI）。但因為有「確認步驟」，使用者可以手動修正。

## 三、資料模型（簡化版）

### 3.1 主管欄位設計 — 從「雙主管」改為「彈性評分者」

既然雙主管是因為去年特殊的合併情況，明年可能只有一位主管，設計上改為彈性欄位：

```sql
-- PRP 主表（每人每期一筆）
CREATE TABLE public.prp_records (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID REFERENCES public.profiles(id) ON DELETE CASCADE,
    period TEXT NOT NULL,                    -- e.g. "2025"
    department TEXT,
    job_title TEXT,
    employee_code TEXT,
    
    -- 綜合考核
    overall_self_summary TEXT,
    overall_manager_comments JSONB,          -- [{ label: "原主管", comment: "..." }, { label: "新主管", comment: "..." }]
    final_rating TEXT,                       -- S/A/B/C/D
    
    -- 面談紀錄
    interview_notes TEXT,
    
    -- 匯入相關
    source TEXT DEFAULT 'import',
    raw_document TEXT,                       -- 原始 Markdown（備份）
    
    created_at TIMESTAMPTZ DEFAULT now(),
    updated_at TIMESTAMPTZ DEFAULT now()
);

-- PRP 明細表
CREATE TABLE public.prp_items (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    prp_record_id UUID REFERENCES public.prp_records(id) ON DELETE CASCADE,
    
    item_type TEXT NOT NULL,                 -- 'kpi' | 'core_competency'
    item_label TEXT NOT NULL,                -- e.g. "KPI 1" 或 "團隊合作"
    importance INTEGER,                      -- 1-3 (可為空，核心職能有時沒標)
    
    self_description TEXT,
    
    -- 彈性評分者 (支援 1~N 位主管)
    evaluations JSONB,                       -- [{ label: "原主管", comment: "...", score: 84 }, { label: "新主管", comment: "...", score: 84 }]
    
    item_rating TEXT,                        -- S/A/B/C/D (行級評等，可為空)
    sort_order INTEGER DEFAULT 0,
    created_at TIMESTAMPTZ DEFAULT now()
);
```

> [!TIP]
> 用 JSONB 存評分者清單，明年只有一位主管時就只有一筆 `[{ label: "主管", comment: "...", score: 85 }]`，完全不影響 schema。

## 四、AI 合成引擎

### 4.1 資料融合架構

```
┌──────────────────────────────────────────────────────────────┐
│                   AI 年度成長總結引擎                          │
│                                                              │
│  Input A: 360 同儕回饋 (Choco360 既有資料)                    │
│  ├── 維度量化分數 (團隊合作 4.3, 問題解決 4.0, 溝通表達 3.5)   │
│  ├── 質性文字 (start/stop/continue + 文字題)                  │
│  └── 維度定義與 purpose (從 questionnaire.dimensions 取得)     │
│                                                              │
│  Input B: PRP 主管考核 (新匯入資料)                            │
│  ├── KPI × N (自述 + 主管評語 + 分數 + 重要度)                 │
│  ├── 核心職能 × 2~5 (自述 + 主管評語 + 分數)                   │
│  ├── 綜合考核評語                                             │
│  └── 最終等第 S/A/B/C/D                                      │
│                                                              │
│  Alignment Layer: Choco360 維度作為共通語言                    │
│  ├── AI 自動將 PRP 核心職能映射到 Choco360 維度                │
│  │   例: PRP「團隊合作」→ 360「團隊合作」                      │
│  │   例: PRP「追求卓越」→ 360「問題解決」(語意最近)             │
│  └── KPI 的行為描述也會被 AI 與維度做語意關聯                   │
│                                                              │
│  ══════════ Output ══════════                                │
│  ├── ⚡ Superpowers (≤3): 多源交叉驗證的顯性優勢               │
│  ├── 🌱 Opportunities (2~3): 需改善的成長領域                  │
│  ├── 📝 Year Summary: 年度成長總結 (200字)                    │
│  └── 🎯 Quests: 建議的下一步發展行動 (3項)                    │
└──────────────────────────────────────────────────────────────┘
```
